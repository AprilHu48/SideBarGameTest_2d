# AGENTS

本文件定义本项目中 AI Agent（如 Codex、Copilot、Cursor Agent）的默认协作规范。

理解项目从README.md开始。

## 1. 目标

- 在不破坏现有功能的前提下，快速完成可验证的增量开发。
- 优先保证可读性、可维护性和可回滚性。
- 产出应符合 Unity 项目常见工程实践。

## 2. 项目范围

- 引擎：Unity（C#）
- 主要目录：
  - `Assets/`：游戏逻辑、资源与编辑器脚本
  - `Packages/`：包依赖与清单
  - `ProjectSettings/`：项目配置
- 默认只修改与任务直接相关的文件；避免无关重构。

## 3. 工作方式

1. 先理解需求与上下文，再动手修改代码。
2. 优先复用现有模式与工具函数，避免引入不必要抽象。
3. 每次改动保持最小闭环：实现 + 基本验证。
4. 不能验证时明确说明未验证项与潜在风险。

## 4. 代码约定（默认）

- 使用 C# 与 Unity 官方推荐写法。
- 命名清晰，避免单字符变量（循环计数器除外）。
- 公共 API 保持稳定，非必要不改签名。
- 新增注释应解释“为什么”，而不是重复“做了什么”。
- 避免一次性大规模格式化，减少无关 diff。
- 编写清晰明了的注释，具体到每个类以及其函数字段的说明。

## 5. Unity 特定约定

- 运行时代码放在 `Assets` 合理子目录；编辑器代码放在 `Editor` 目录下。
- 新增脚本时关注命名空间一致性。
- 修改序列化字段时评估对已有场景/预制体的影响。
- 若涉及 ScriptableObject、Prefab、Scene，说明兼容性影响。

## 6. 提交与变更说明

- 一个任务一个主题，保持变更聚焦。
- 说明中应包含：
  - 改了什么
  - 为什么这样改
  - 如何验证
  - 已知限制/后续建议（如有）

## 7. 禁止事项

- 未经明确要求，不执行破坏性操作（如批量删除、强制回滚）。
- 不修改无关模块来“顺手优化”。
- 不引入未经验证的重量级依赖。

## 8. 默认交付清单

- 代码改动（最小必要范围）
- 简要验证结果
- 风险与后续建议（如需要）

## 9. 测试与任务同步规则

- 不同模块的测试说明应放置在其对应模块实现说明文档目录下的 `Test` 子文件夹。
- 每完成任务清单中的一个任务，应及时为该任务打上完成标记。

## 项目说明文档

- 按系统目标、架构分层、核心模块、数据流、工具、测试、边界与扩展来组织
- 合理拆分文档结构，不要只放在父级文件夹根目录或一个文本下，将不同模块拆分到不同子文件夹
- 适合后续长期维护和新成员快速上手

---

如无额外说明，以上规则作为本仓库 AI Agent 的默认行为准则。
